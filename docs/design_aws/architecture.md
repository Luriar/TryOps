# TryOps Architecture v1 — Stage 1·2 매장 MVP·확장 설계

> 본 문서는 TryOps 시스템의 **Stage 1·2 (매장 MVP + 확장 단계) 아키텍처 설계**다.
>
> **Stage 0 (개념 검증)** 은 별도 문서 `concept_validation_spec.md` 참조. **Stage 0 통과 후에만 본 설계 진입**.
>
> **설계 목표**: PoC 매장 1~3개부터 매장 100개까지 매장당 한계비용 7.5만원/월 이하 유지.
>
> **본 v1의 한계**: 실제 AWS 계정·매장 환경 미실측. 비용은 공개 가격표 + 추정. Stage 0 완료 후 PoC Week 1~4에 실측 후 v2 정밀화.
>
> 신뢰도 표기:
> - 🟢 1차 출처 검증 (AWS 공식 가격표, 제품 가격)
> - 🟡 합리적 추정 (검증 데이터 기반)
> - 🔴 가설 (PoC 단계 검증 필요)

## 0. Executive Summary — 30초

TryOps 시스템은 **매장 측 ESP32-S3 노드 → 매장 게이트웨이 → AWS Firehose → S3 Parquet → Athena/Lambda** 의 단순 구조로 시작한다. PoC 단계 매장 1~3개의 월 AWS 비용은 **약 $20~30**, 매장 100개 도달 시 매장당 한계비용 **약 $50~60** (월 약 6.5~7.8만원) 유지 가능. 

핵심 설계 결정 3가지:
1. **Kinesis Data Streams 대신 Kinesis Firehose** — 단순 S3 적재용, 시간당 비용 없음 + GB당 30% 저렴
2. **Lambda + Athena 조합** — 서버 관리 없음, 매장 수 비례 비용
3. **Polars + DuckDB on EC2 t4g.small** — Phase 1·2에는 PySpark 불필요, 단일 인스턴스로 충분

## 2. 핵심 설계 결정 5가지

### 결정 1: Kinesis Data Streams → Kinesis Firehose 변경

**기존 가정** (product_strategy.md): Kinesis Data Streams on-demand

**v1 설계 변경**: **Kinesis Firehose**

**근거**:
- 🟢 Kinesis Data Streams on-demand: $0.04/GB + **per-stream $0.04/시간** (스트림당 월 $29)
- 🟢 Kinesis Firehose: $0.029/GB, **스트림 시간 비용 없음**
- 본 사용 케이스는 *"매장 → S3 적재"* 단순. Firehose가 본질적 정답
- 출처: AWS Kinesis 공식 가격표, Wring Blog 2026.03

**손실**:
- Firehose는 **60초 버퍼링 후 적재** (실시간 < 60초 알람 어려움)
- Assistance Need 실시간 알람은 *"매장 게이트웨이에서 직접 처리"* 로 우회

### 결정 2: 매장 게이트웨이가 *"가장 큰 의사결정자"*

**핵심**: 매장 측에서 *"무엇을 집계하고 무엇을 보낼지"* 결정. **AWS로 보내기 전 90% 줄임**.

**매장 게이트웨이 역할**:
- ESP32-S3 노드 4~6개의 CSI raw 수집 (초당 약 10KB × 6 = 60KB/초)
- 1분 단위 윈도우 집계 (활동 강도 점수, 점유 추정)
- RFID·POS webhook 통합
- 5분 배치로 AWS 전송 (1MB 미만)
- 7일 로컬 버퍼 (네트워크 장애 대비)

**하드웨어**:
- Raspberry Pi 4B 8GB (약 12만원)
- microSD 64GB (약 1.5만원)
- 케이스·전원: 약 3만원
- **매장 게이트웨이 총 약 16만원** (1회성)

**SW 스택**:
- Ubuntu 22.04 Server (ARM64)
- Python 3.11 + Polars (집계)
- SQLite (로컬 버퍼)
- systemd (서비스 관리)

### 결정 3: Polars + DuckDB on EC2 t4g.small

**기존 가정** (product_strategy.md): Phase 3에 PySpark on EMR Serverless 마이그레이션

**v1 설계 명확화**: Phase 1·2는 **EC2 t4g.small + Polars + DuckDB로 충분**.

**근거**:
- 매장 100개 × 일 1MB = 일 100MB = 월 3GB. **단일 t4g.small (2vCPU, 2GB RAM, ARM Graviton2)** 로 충분
- 🟢 EC2 t4g.small 월 비용: 약 $15 (서울 리전, on-demand)
- Polars + DuckDB 조합이 PySpark 보다 *"단일 노드 효율"* 압도적
- API 패턴이 Spark와 호환되도록 작성 → Phase 3 마이그레이션 자연스러움

**왜 Lambda 만으로 부족한가**:
- Airflow DAG가 필요 (10분/1시간/일 단위 ETL 스케줄링)
- Lambda 15분 제한 + 콜드 스타트 부담
- DuckDB는 단일 인스턴스 인메모리가 가장 빠름

### 결정 4: S3 Parquet 파티셔닝 구조

**파티션 키**:
- `store_id` (매장 식별)
- `date` (날짜)

**버킷 구조**:
```
s3://tryops-data/
├── raw/
│   └── store_id=X/date=YYYY-MM-DD/HH/data.parquet
├── mart/
│   ├── sessions/store_id=X/date=YYYY-MM-DD/data.parquet
│   ├── joint_signals/store_id=X/date=YYYY-MM-DD/data.parquet
│   └── flow_signals/store_id=X/date=YYYY-MM-DD/data.parquet
└── reports/
    └── brand_id=Y/month=YYYY-MM/data.parquet
```

**근거**:
- 🟢 Athena Parquet + 파티셔닝으로 비용 85~95% 절감 (Cloud Burn 2026.04)
- 매장당 / 일자별 쿼리가 본사 대시보드의 90%
- 본사 보고서는 월 단위 집계로 별도 마트

**S3 저장 비용**:
- 🟢 S3 Standard: $0.023/GB-month (서울)
- 매장 100개 × 월 3GB = 300GB → **월 약 $7**

### 결정 5: 본사 대시보드 = Cloudflare Pages + Lambda

**프론트엔드**:
- **Cloudflare Pages** (Next.js static export)
- 🟢 무료 (10만 요청/일까지)
- CDN 자동, 글로벌 빠름
- 본사·매장 대시보드 같은 SPA 분기

**백엔드**:
- API Gateway + Lambda (Python 3.11)
- Athena 쿼리 결과 캐싱 (Lambda 메모리 또는 ElastiCache)
- 🟢 API Gateway: $3.50/백만 요청, Lambda: $0.20/백만 요청
- 본사당 일 100 요청 가정 → 매장 100개당 월 약 $1

**인증**:
- **Cognito User Pools** — 본사·매장 매니저 멀티 테넌트
- 🟢 월 5만 MAU까지 무료
- 본사당 사용자 10~30명 가정 → 무료 한도 내 (본사 1,000개까지)

## 3. 본 설계의 가장 큰 실패 위험 — 통합 부담 (v2 셀프리뷰 본질 3 해소)

본 설계의 *"가장 싸게"* 와 *"단순 아키텍처"* 를 무너뜨릴 수 있는 가장 큰 위험은 **외부 시스템 통합의 누적 부담**.

### 3.1 통합 대상 시스템 3종

**A. RFID 솔루션사 통합** (Funnel Module 필수):
- 셀메이트 — 한국 패션 RFID 1위, 자체 webhook API
- idro — 차세대 RFID 솔루션, REST API
- 하이태그랩 — 패션 특화, FTP 기반 (legacy)
- ID테크 — 글로벌 진출, OAuth 2.0
- 기타 본사 자체 RFID 시스템 (F&F · 코오롱FnC 등)

**B. POS 솔루션사 통합** (Funnel Module 필수, Flow Module 선택):
- 토스플레이스 — D2C 본사 보편, 표준 REST API
- NHN KCP — 대형 본사 보편, SOAP/REST 혼재
- NICE 페이먼츠 — 결제 인프라, 별도 API
- 금융결제원 — 일부 본사, 표준 인터페이스
- 자체 POS (F1 FASHION 결 본사) — 본사별 맞춤 통합

**C. 본사 ERP·SCM 통합** (Funnel Enterprise ARR 필수):
- SAP S/4 HANA — F&F·삼성물산 등 대기업 보편
- Oracle EBS — 일부 대기업
- 자체 시스템 — K-D2C 본사 보편 (안다르·젝시믹스 등)

### 3.2 통합 부담 누적 추정

매장 수별 누적 통합 작업 (`tech_stack_evolution.md` 섹션 6.5.4 참조):

| 매장 수 | 본사 수 | 누적 RFID 통합 | 누적 POS 통합 | 누적 ERP 통합 | 누적 통합 인월 |
|---|---|---|---|---|---|
| 1~10 | 1~2 | 1~2 | 1~2 | 0 | 약 3~6 인월 |
| 10~100 | 3~10 | 3~5 | 3~5 | 1~2 | 약 15~30 인월 |
| 100~500 | 10~30 | 5~8 | 5~8 | 3~5 | 약 50~100 인월 |
| 500~1,000 | 30~50 | 8~12 | 8~12 | 5~10 | 약 100~200 인월 |

🔴 **누적 통합이 PoC 1년차 1.5억 → 실제 3~5억 가능성의 주범**.

### 3.3 위험 완화 전략 3가지

**전략 1: 표준 Adapter 인터페이스 (Phase 2 우선 작업)**

```python
# 표준 인터페이스 정의
class RFIDAdapter(ABC):
    @abstractmethod
    def parse_event(self, raw_payload: dict) -> RFIDEvent:
        """솔루션사별 raw payload → 표준 RFIDEvent 변환"""
        pass
    
    @abstractmethod
    def authenticate(self) -> str:
        """솔루션사 인증 (API key·OAuth·자체)"""
        pass

# 솔루션사별 구현
class SelmateAdapter(RFIDAdapter): ...
class IdroAdapter(RFIDAdapter): ...
class HighTagLabAdapter(RFIDAdapter): ...
```

표준 인터페이스 구축 → 신규 솔루션사 통합 1~2주 → **2~3일로 단축 목표**.

**전략 2: 본사 자체 통합 위임**

본사 ERP·SCM 통합은 *"TryOps가 표준 webhook 제공, 본사가 자체 통합 구현"* 으로 위임:
- TryOps → 본사 webhook 송신 (표준 스키마)
- 본사 시스템팀이 webhook 수신 통합 (본사 자체 부담)
- TryOps 통합 작업 0건

본사가 *"본사 자체 통합 어렵다"* 거부 시 *"통합 컨설팅 별도 비용"* 으로 옵션 제공.

**전략 3: 통합 파트너십 (Phase 3+)**

매장 100개 도달 후 솔루션사와 공식 파트너십:
- 셀메이트 공식 파트너 인증 → 셀메이트가 자체 통합 모듈 제공
- 토스플레이스 마켓플레이스 등록 → 본사 자동 통합

**파트너십 협상 비용**: 시간 (3~6개월) + 수익 분배 (TryOps 매출의 10~20%).

### 3.4 본 위험의 시장 영향

본 통합 부담이 본 설계의 *"실패 시나리오"* 중 가장 큰 것:
- product_strategy.md 섹션 15 실패 시나리오에 *"실패 10: 통합 부담 누적"* 추가 권고
- 영업 시 *"매장 N개 도달 시 통합 부담 폭증"* 의 본사 안내 매뉴얼 필요

🔴 **PoC Week 1~4 우선순위**: 셀메이트·토스플레이스 표준 통합 1개씩 검증 후 표준 Adapter 인터페이스 v1 작성. **표준 Adapter가 본 설계의 가장 큰 운영 자산**.

## 5. 데이터 흐름 5단계

### 3.1 Stage 1: 매장 → 매장 게이트웨이 (실시간)

```
ESP32-S3 노드 (피팅룸당 1~2개)
  ↓ MQTT over WiFi (내부 매장 WiFi)
매장 게이트웨이 (Raspberry Pi)
  - Mosquitto MQTT broker
  - CSI raw 적재 (SQLite)
```

데이터 양: 노드당 초당 약 10KB → 매장 총 60~100KB/초

### 3.2 Stage 2: 매장 게이트웨이 집계 (1분 단위)

```
매장 게이트웨이 cron (1분):
  1. 지난 1분 CSI raw 로드
  2. Polars 집계:
     - 활동 강도 점수 (0~1 정규화)
     - 점유 추정 (0/1/2+)
     - 움직임 패턴 분류
  3. RFID 이벤트와 시간축 결합
  4. POS webhook 데이터 통합
  5. 결과 1분 윈도우 → SQLite 누적
```

**핵심**: 이 단계에서 **데이터를 99% 줄인다**. 60KB/초 raw → 1KB/분 집계.

### 3.3 Stage 3: 매장 → AWS (5분 배치)

```
매장 게이트웨이 cron (5분):
  1. 지난 5분 집계 데이터 로드 (약 5KB)
  2. API Gateway PUT 요청
  3. 성공 시 SQLite 7일 보관 (장애 대비)
  4. 실패 시 재시도 (exponential backoff)
```

데이터 양: **매장당 5분당 약 5KB = 일 약 1.5MB = 월 약 45MB**.

매장 100개: 월 약 4.5GB. **AWS 전송 비용 무시 수준**.

### 3.4 Stage 4: AWS Ingestion → S3 (실시간 → 1분)

```
API Gateway → Lambda (Ingest)
  ↓
Kinesis Firehose (60초 버퍼링)
  ↓
S3 Parquet 적재
  - 파티션: store_id, date, hour
  - 압축: Snappy (Parquet 기본)
```

🟢 Firehose: $0.029/GB → 매장 100개 월 4.5GB → **월 $0.13** (사실상 무료)

### 3.5 Stage 5: ETL → Mart (배치)

```
EventBridge cron:
  - 10분: 실시간 알람용 마트 (Assistance Need 등)
  - 1시간: Joint Signal 5종 계산
```

### 4.7 매장당 데이터량 1.5MB/일 가정의 산식 (비판 1 부분 해소)

본 v1의 모든 AWS 비용이 *"매장당 일 1.5MB"* 가정에 걸려 있다. 본 추정의 산식:

**매장 게이트웨이 → AWS 5분 배치 페이로드 산식**:
- 1분 CSI 집계 1개: activity_score(4B) + occupancy(4B) + movement_pattern(16B) + timestamp(8B) + 메타(약 20B) = **약 50~100B**
- 피팅룸 5개 × 5분 = 25개 1분 집계 = **약 1.25~2.5KB**
- RFID 이벤트 (5분 평균 20개 × 100B) = **약 2KB**
- POS 이벤트 (5분 평균 5개 × 200B) = **약 1KB**
- 헤더·메타: **약 0.5KB**
- **5분 배치 합계: 약 5KB** (보수적 추정)

일 기준: 5KB × 288 (5분 간격) = **약 1.44MB/일** ≈ 가정 1.5MB

🟡 **추정의 한계**: 위 산식은 *"정상 운영 케이스"*. 실제 흔들릴 수 있는 요인:
- 매장 피크 시간 데이터 변동성
- 디버깅 로그 포함 시 2~5배 증가 가능
- RFID·POS 솔루션사별 페이로드 차이
- → PoC Week 1 실측 절대 필수

### 4.8 Athena 쿼리 패턴 산식 (비판 2 부분 해소)

본 v1의 Athena 비용 *"매장 100개·본사 5개 시 월 $1.50"* 의 산식:

**본사 사용자별 쿼리 빈도 추정** (`personas_extended.md` v4 페르소나 기반):
- 데이터팀장 (김지현·박정우): 일 10~30회 깊은 쿼리, 평균 1GB scanned
- 머천다이저 (박서연): 일 20~50회 SKU별 쿼리, 평균 200MB scanned
- 매장 매니저 (정민호): 매장당 일 50~100회 실시간 폴링, 평균 50MB scanned (캐시 우회 시)
- 본사 임원·CFO: 주 1~5회 보고서 쿼리, 평균 100MB scanned

**본사당 일 쿼리 비용 추정 (캐시 우회 worst case)**:
- 데이터팀장 30 × 1GB = 30GB
- 머천다이저 50 × 200MB = 10GB
- 매장 매니저 100 × 매장 평균 10 × 50MB = 50GB
- 임원 5 × 100MB = 0.5GB
- 본사당 일 합계 = **약 90GB scanned** 

본 v1 가정 (본사당 일 30회) 대비 **약 3배 더 많음**. 캐싱 없을 시 비용이 v1 가정의 3~5배 가능.

**대응 (캐싱 강화)**:
- Lambda 메모리 캐싱 (1시간 TTL)
- ElastiCache Redis 도입 (Phase 3)
- 매장 매니저 폴링은 매장 게이트웨이에서 직접 제공

캐싱 적용 시 본사당 일 scanned 90GB → **약 20GB** (78% 절감). 본 v1 가정 30GB와 근사.

🔴 본 산식은 PoC 단계 본사 실제 사용 패턴 측정으로 정밀화 필요.

4. **모바일 반응형 (Phase 2+)**: 매장 매니저는 매장 내 태블릿·폰 사용
5. **다크 모드 (선택)**: 매장 백오피스의 어두운 환경 고려

### 6.5 UI 구현 가이드 (v2 셀프리뷰 형식 8 해소)

본 와이어프레임을 실제 코드로 만들 때의 구현 가이드.

**기술 스택**:
- Next.js 14 (App Router) + TypeScript
- Tailwind CSS + shadcn/ui (컴포넌트)
- Recharts (그래프) 또는 Tremor (대시보드 전용)
- TanStack Query (API 통합 + 캐싱)
- Zustand (전역 상태)

**페르소나별 화면 컴포넌트 구조**:
```
apps/web/src/
├── app/
│   ├── (dashboard)/
│   │   ├── brand/insights/    # 데이터팀장 메인 (섹션 6.1)
│   │   ├── brand/sku/[id]/    # 머천다이저 SKU 상세 (섹션 6.2)
│   │   └── store/[id]/        # 매장 매니저 실시간 (섹션 6.3)
│   └── api/
├── components/
│   ├── ui/             # shadcn/ui
│   ├── charts/         # Recharts 래퍼
│   └── alerts/         # 실시간 알람
└── lib/
    ├── api.ts          # Lambda 호출
    └── auth.ts         # Cognito
```

**반응형 디자인** (매장 매니저용):
- 모바일 (320~640px): 1열, 카드
- 태블릿 (640~1024px): 2열, 피팅룸 그리드
- 데스크탑 (1024px+): 3~5열, 전체 매장

**거부형 페르소나 (강희정) 권한 구조**:
```typescript
// 매장 매니저 본인 대시보드 권한
{
  brand_admin_can_see: false,   // 본사 임원이 매니저 개별 데이터 못 봄
  store_manager_can_see: true,  // 매니저 본인만
  metrics: ["my_assistance_contribution", "my_conversion_rate"]
}
```

**Cognito role 매핑**:
- `data_lead` → 본사 메인 + 모든 매장
- `merchandiser` → SKU 상세 + 본사 메인 (제한)
- `store_manager` → 매장 실시간 + 본인 대시보드만
- `viewer` → 본사 메인 읽기 전용

🔴 디자이너 협업 + 프론트엔드 4~8주 작업 가정. 실제 디자인 시 와이어프레임 다듬기 필수.

### 6.6 매장 100개 동시 운영 시 부하 분산 다이어그램 (v2 셀프리뷰 형식 7 해소)

본 v2 섹션 1 시스템 전체 구성도는 *"매장 1개 관점"*. 실제 매장 100개 동시 운영의 부하 분산·격리 시각화.

```
┌─────────────────────────────────────────────────────────────┐
│ 매장 측 (100개 게이트웨이 동시 운영)                          │
│                                                              │
│   [매장 1]   [매장 2]   [매장 3]   ...   [매장 100]            │
│   GW 1      GW 2      GW 3              GW 100              │
│      │        │        │                  │                  │
│      └────────┴────────┴──────────────────┘                  │
│                          ↓                                    │
│         (각 매장 5분 배치 약 5KB 송신)                          │
└──────────────────────────┼───────────────────────────────────┘
                           │ HTTPS (전체 5분당 약 500KB 동시)
                           ↓
┌──────────────────────────────────────────────────────────────┐
│ AWS (자동 스케일링)                                            │
│                                                               │
│   API Gateway (한도 10,000 RPS)                              │
│     매장 100개 = 0.3 RPS 총합 → 한도 0.003% 사용              │
│                          ↓                                    │
│   Lambda Ingest (동시 1,000 한도)                             │
│     매장 100개 = 동시 0.5 호출 → 한도 0.05% 사용              │
│                          ↓                                    │
│   Firehose Stream (5MB/초 자동 스케일)                        │
│     매장 100개 5분당 500KB = 1.7KB/초 → 한도 0.03% 사용       │
│                          ↓                                    │
│   S3 (사실상 무제한)                                          │
│     매장 100개 일 4.5GB → 매장당 store_id partition 격리      │
│                                                               │
│   EC2 t4g.small ETL (병목 가능 지점!)                         │
│     Polars 1시간 DAG: 매장당 1초 처리 가정 시 100초            │
│     DAG 1시간 간격 → 충분히 여유                              │
│     Phase 3 도달 시 EMR Serverless 진화                       │
│                                                               │
│   Athena (동시 쿼리 5 한도)                                   │
│     본사 5개 동시 시 가끔 큐잉                                  │
│     한도 증가 신청 가능 (Phase 3)                              │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

**핵심 부하 격리 메커니즘**:

| 격리 단위 | 방법 | 한 본사 부하 spike의 영향 |
|---|---|---|
| API Gateway | per-brand throttle | 다른 본사 영향 0 |
| Lambda | 본사별 reserved (Phase 3) | 분리 가능 |
| S3 | brand_id partition | 별도 prefix, 영향 0 |
| Athena | 본사별 Workgroup | 동시 쿼리 한도 분리 |
| DynamoDB | per-brand partition key | 자동 분산 |

🟢 본 부하 분산이 *"본사 1곳 부하 spike가 다른 본사 영향 0"* 의 멀티테넌트 격리. `security_design.md` 섹션 6 통합 매트릭스 참조.

**Phase 1·2 (매장 200개까지) 진짜 병목**:

| 병목 | 발생 시점 | 대응 |
|---|---|---|
| EC2 t4g.small CPU | 200~300매장 | EC2 m6g.large 스케일업 또는 EMR Serverless |
| Athena 동시 쿼리 5 | 본사 5+ 동시 쿼리 시 | 캐싱 강화 또는 한도 증가 |
| Lambda 동시 1,000 | 본사 1,000+ | 한도 증가 신청 |
| Cognito MAU 5만 | 본사 1,000+ | Identity Pools 분리 |

본 4가지 모두 `tech_stack_evolution.md` Phase 3 진화 트리거 명시.

🟢 **Phase 1·2 (매장 200개까지) 는 자동 스케일링 + AWS 한도 내에서 안정 동작**.

🔴 UI/UX 정밀 명세는 별도 디자인 문서에서. 본 v2는 골격만.

## 9. Phase 1·2의 한계 — Phase 3 진화 트리거

본 v1 설계는 매장 100~200개까지 안정 동작. 다음 트리거 도달 시 Phase 3 진화 필요.

### 5.1 Phase 3 진화 트리거

| 트리거 | 의미 | 대응 |

## 10. 보안·법적 준수 설계

`legal_review.md` v1의 6개 법령 준수를 시스템 설계에 반영.

### 6.1 개인정보 격리

- 매장 게이트웨이에서 **개인 식별 데이터 제거** (POS의 카드번호·전화번호 등)
- AWS 전송은 *"매장 단위 익명 집계"* 만
- raw CSI는 매장 게이트웨이 7일 후 자동 삭제
- AWS S3 적재 데이터는 모두 *"매장 단위 익명"*

### 6.2 데이터 보존 정책

| 데이터 | 위치 | 보존 기간 |
|---|---|---|
| CSI raw | 매장 게이트웨이 SQLite | 7일 |
| 매장 1분 집계 | 매장 게이트웨이 SQLite | 30일 |
| AWS S3 raw | S3 Standard | 90일 |
| AWS S3 mart | S3 Standard | 1년 (집계 후 익명) |
| AWS S3 reports | S3 Standard | 영구 (본사 보유) |

### 6.3 암호화

- 매장 → AWS: TLS 1.3 (HTTPS)
- S3 at-rest: SSE-S3 (기본)
- RDS at-rest: KMS 암호화
- 매장 게이트웨이 SQLite: SQLCipher (선택)

### 6.4 의료기기법 회피

- 호흡·심박 데이터는 **숫자로 표시 안 함**
- *"활동 강도 점수"* (0~1) 로만 표기
- 응급 감지·건강 모니터링 기능 일절 없음

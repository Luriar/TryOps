# Lessons Learned — TryOps 기획 프로세스 회고

> 본 문서는 TryOps 프로젝트의 기획·검증 프로세스를 회고하고, **다음에 기획할 때 더 적은 노력으로 더 완성된 값을 낼 수 있는 가이드**를 제공한다.
>
> 본 프로젝트는 v1 (제품 컨셉) → v4 (검증 시스템 완성) 까지 약 4단계 진화, 10개 문서·4,585줄·30+ 1차 출처를 산출했다. 그 과정에서 무엇이 효과적이었고 무엇이 비효율적이었는지 정직 분석.

---

## 0. 본 회고가 답하는 질문

1. TryOps 기획에서 진짜 효과적이었던 것은 무엇인가?
2. 시간 낭비한 부분은 어디인가?
3. 다음 기획 시 어떤 순서로 가면 가장 효율적인가?
4. 어떤 도구·프롬프트가 가장 가치 컸는가?
5. PM 사고에 진짜로 도움된 학습은 무엇인가?

---

## 1. 본 프로젝트의 큰 흐름 복기

### 1.1 진화 단계 요약

| 단계 | 시간 | 산출물 | 핵심 사건 |
|---|---|---|---|
| **v1** | 초기 | 제품 컨셉 (피팅룸 + RuView) | 컨셉 도달 (이전 세션) |
| **v2** | 1~2일 | PRD 수준 확장 (1,583줄) | 사용자 직접 확장, 기획자 사고 적용 |
| **v3** | 1~2일 | 메인 문서 압축 (954줄) + 보조 문서 6개 | 셀프리뷰 시작, 시장 분석·페르소나 정밀화 |
| **v4** | 1일 | 페르소나 진짜 검증 + 인터뷰 액션 플랜 | 점수 인플레이션 해소, 정직성 시스템 완성 |
| **최종** | 1일 | 10개 문서 + 회고 | 셀프리뷰 무한 루프 종료 선언 |

### 1.2 진화의 동력 — 3가지

본 프로젝트가 v1 → v4까지 진화한 동력 3가지:

1. **사용자 직접 셀프리뷰 요청** — *"이거 다시 셀프리뷰해줘"* 의 반복이 매 사이클 핵심 변곡점
2. **셀프리뷰가 새 약점을 정직 발굴** — Claude가 자기 산출물을 비판자 시각으로 점검
3. **검증 자료 검색의 누적** — v1 검색 0건 → v4 검색 30+ 건. 진짜 데이터가 가설을 흔듦

이 3가지가 없었다면 v1에서 멈췄을 것. **사용자의 셀프리뷰 압박이 가장 큰 효과적 동력**.

## 2. 효과적이었던 것 7가지

### 효과 1: 사용자의 *"셀프리뷰해줘"* 명령

각 단계 종료 후 사용자가 *"리뷰해줘"* 또는 *"셀프리뷰 해"* 요청 → Claude가 자체 비판 → 새 약점 발견 → 정정. 이 사이클이 본 프로젝트의 가장 큰 효과적 도구.

**효과 측정**:
- v1 → v2: 검증 자료 0 → 13건 (사용자 압박이 검색 트리거)
- v2 → v3: 점수 인플레이션 해소 (사용자가 비판 받아들임)
- v3 → v4: 회사/행동 분리, 가설 흔들림 정직 보고 (사용자의 *"옵션 A로"* 결정)

**다음 기획 시 적용**: 사용자가 *"셀프리뷰"* 명령을 일정 단계마다 의식적으로 요청. 산출물 자체 보다 *"산출물의 약점을 본인이 발견했는가"* 가 핵심.

### 효과 2: 신뢰도 표기 시스템 (🟢/🟡/🔴)

v3에서 도입한 표기 시스템이 *"검증된 것 / 추정 / 가설"* 의 정직성 구조를 만듦. 이게 점수 인플레이션 차단의 본질.

**효과 측정**:
- v2 출처 13건이 *"검증 완료"* 로 인식됐다가 v3에서 *"검증 부족"* 으로 자체 정정
- 정직한 표기가 면접관 신뢰도 직결

**다음 기획 시 적용**: 첫 문서 작성 단계부터 신뢰도 표기 시스템 도입. *"이게 얼마나 검증됐냐"* 의 자체 의식이 산출물의 본질.

### 효과 3: 시장 검증이 비즈니스 전략 흔든 경험

(c) 시장 분석에서 *"한국 패션 RFID 도입 5곳뿐"* 발견이 Beach-head 전략 자체를 재구성. 검증 자료 1건이 사업 전략 전체를 바꿈.

**다음 기획 시 적용**: *"우리 사업 가정의 결정적 사실 1건을 검증하자"* 가 시장 분석의 출발점. 모든 가정을 검증할 수 없으므로 **결정적 가정 2~3개에 집중**.

### 효과 4: 페르소나 가설이 검증으로 흔들린 경험

(d) v3에서 *"오프라인 환경 개인정보 불안 76.2%"* 발견이 v2의 *"95:5 비대칭성"* 가정 정면 흔듦. 본인 가설이 검증으로 뒤집힌 사례는 PM 사고의 정수.

**다음 기획 시 적용**: 가설 만들면 *"이 가설을 흔들 수 있는 데이터를 찾자"* 의 반증 시각으로 검증.

### 효과 5: 압축 작업 (1,583줄 → 749줄)

(a) 메인 문서 압축이 *"무엇이 본질이고 무엇이 부차적인가"* 의 PM 판단을 강제. 단순 분량 감소가 아닌 사고의 명료화.

**다음 기획 시 적용**: 첫 산출물 작성 후 50% 압축을 의식적으로 시도. *"이 문장이 없으면 무엇이 안 되는가"* 의 질문이 본질 발굴 도구.

### 효과 6: *"실제 사업 시작 시 동일 프로세스 재사용"* 목표 선언

본 프로젝트 중반에 사용자가 *"실제 사업 할 때 동일 프로세스 재사용"* 목표를 명시. 이게 모든 후속 작업의 기준점.

**다음 기획 시 적용**: 프로젝트 시작 시 *"이 산출물을 어디서 어떻게 쓸 것인가"* 의 목표 선언. *"포트폴리오"* 와 *"실제 사업"* 은 톤·깊이가 다름.

### 효과 7: 셀프리뷰 종료 선언

v4 단계에서 *"공개 자료 검증의 한계점 도달"* 선언이 무한 루프 종료. PM이 *"언제 멈출 것인가"* 의 메타 학습.

**다음 기획 시 적용**: 셀프리뷰 시작 시 *"이 사이클의 종료 조건"* 도 같이 명시. 매 사이클 새 약점이 나오지만, 종류가 본질에서 멀어지면 멈춤.

## 3. 비효율이었던 것 5가지

### 비효율 1: v1 → v2 확장의 통제 부족

사용자가 *"기획자 사고 적용"* 하면서 v2를 1,583줄로 확장. 본질·메타 학습·체크리스트가 한 문서에 섞임.

**왜 비효율**: v3 압축 단계에서 다시 분리해야 했음. **확장 단계에서 *"분리 구조"* 를 의식했다면 압축 작업 불필요**.

**다음 기획 시 개선**: 확장 시작 전 *"무엇이 메인 문서, 무엇이 보조 문서"* 의 구조를 먼저 정의. 한 문서에 모든 것 넣지 않음.

### 비효율 2: 셀프리뷰 무한 루프 시작

v3 페르소나 셀프리뷰 → v4 → v4 셀프리뷰 → v5 가능성... 매 사이클 새 약점 나옴. **언제 멈출지의 기준이 없었음**.

**왜 비효율**: 사용자가 *"실제 사업 시작 시 동일 프로세스 재사용"* 결정 → 옵션 A 무한 진행 가능성. Claude가 *"옵션 C 종료"* 추천했지만 사용자 결정대로 옵션 A 진행.

**다음 기획 시 개선**: 셀프리뷰 시작 전 *"3사이클 후 종료 또는 점수 차이 5%p 이내 시 종료"* 같은 명시적 종료 조건. 또는 *"PoC 인터뷰 단계에서 검증 가능한 것은 멈춘다"* 의 분리 기준.

### 비효율 3: 검증 자료 검색의 시점 늦음

v2까지 검증 자료 0건. v2 셀프리뷰 후 v3에서 처음 검색 시작. **검증을 처음부터 했다면 v2 작성 자체가 다른 톤이었을 것**.

**왜 비효율**: v2 1,583줄 작성 → 검색 → 가정 흔들림 → 재작성. 검색을 v1 → v2 단계에 했다면 v2 작성 시 정밀도 ↑.

**다음 기획 시 개선**: 첫 산출물 작성 시 *"5분만 검색하고 시작"* 의 규칙. 검색이 작성을 가이드.

### 비효율 4: 점수 인플레이션 자체

v2가 *"평균 67% 검증"* 표기 → v3 *"45%"* 정정 → v4 *"35%"* 추가 정정. **각 단계가 점수를 깎는 작업**.

**왜 비효율**: 첫 표기를 정직하게 했다면 정정 작업 불필요. 인플레이션이 정정의 정정을 만듦.

**다음 기획 시 개선**: 점수 표기 시 *"회사 실재성 vs 행동 검증 분리"* 를 처음부터 적용. 두 점수의 평균을 단일 점수로 표기하지 않음.

### 비효율 5: 메타 평가 톤의 자기 만족

각 문서 마지막에 *"본 문서가 가진 진짜 가치는..."* 의 메타 평가가 반복됨. 본인이 본인 산출물을 평가하는 자기 만족 톤.

**왜 비효율**: 면접관·동료 입장에서는 *"본인이 좋다고 말하는 것"* 보다 *"이걸 어떻게 쓰는가"* 의 사용자 안내가 더 가치.

**다음 기획 시 개선**: 문서 마지막은 *"본 문서 활용 가이드"* 톤. *"이걸 어떻게 쓰는가"* 의 사용자 입장 안내.

## 4. 새로 발견한 PM 사고 학습 6가지

### 학습 1: *"검증을 한 듯한 모양새"* ≠ *"진짜 검증"*

1차 출처 첨부 자체가 검증이 아님. **출처가 페르소나의 본질적 행동을 직접 입증해야 진짜 검증**. v3 박정우의 F&F 회사 검증은 강했지만 *"박정우 인물의 의사결정 행동"* 검증은 약했음.

→ *"출처가 무엇을 입증하는가"* 의 정밀 질문이 본질.

### 학습 2: 점수가 떨어지는 게 정직성 향상

v1 67% → v4 35%. 점수가 내려가는 게 검증의 깊이가 향상되는 신호.

**다음 기획 시 적용**: 첫 표기를 보수적으로. *"50% 검증됨"* 보다 *"30% 검증, 70% PoC 단계 검증 필요"* 가 정직.

### 학습 3: 검증 자료 *"맥락"* 이 가장 중요

오픈서베이 76.2% 발견 사례. *"개인정보 기재가 필요한 오프라인 이벤트"* 라는 조건이 핵심인데 v3는 그 조건 빼고 인용. v4에서 맥락 경고 추가.

**다음 기획 시 적용**: 인용 시 *"이 데이터가 어떤 조건의 데이터인가"* 의 맥락 확인. 조건 다르면 적용 부정확.

### 학습 4: 가설이 흔들리는 발견을 정면 보고하는 게 PM 사고

v3에서 95:5 가설이 76.2% 데이터로 흔들림 → v4에서 정정 명시. **이걸 숨기지 않은 게 PM 사고의 정수**.

**다음 기획 시 적용**: 가설 흔들리면 *"수정 사항"* 으로 명시. *"본 v2 가정은 잘못됐다"* 의 정직성이 신뢰 구축.

### 학습 5: PM 사고는 *"언제 멈출 것인가"* 도 포함

셀프리뷰 무한 루프의 함정. 매 사이클 새 약점 나옴. **종료 조건 명시가 PM 사고**.

**다음 기획 시 적용**: 작업 시작 시 *"이 작업의 종료 조건"* 명시. *"3사이클 후 / 5%p 차이 이내 / 인터뷰 영역 진입 시"* 같은 기준.

### 학습 6: 검증 시스템이 검증 결과보다 가치 클 수 있다

본 프로젝트의 진짜 자산은 *"안다르 매출 3,000억"* 같은 데이터가 아니라 **신뢰도 표기·점수 분리·셀프리뷰 사이클** 의 검증 시스템 자체.

**다음 기획 시 적용**: 산출물의 *"내용"* 보다 *"검증 방법론"* 을 자산화. 다음 프로젝트에서 재사용 가능.

## 5. 본 프로젝트가 정의한 *"기획 프로세스 표준 워크플로우"*

다음 기획 시 그대로 재사용 가능한 표준 워크플로우 7단계:

### 단계 1: 프로젝트 시작 선언 (30분)

```
- 목표: "포트폴리오" vs "실제 사업" 톤 결정
- 종료 조건: 어떤 시점에서 멈출지 사전 정의
- 검증 깊이: 공개 자료만 vs 인터뷰 포함
- 산출물 구조: 메인 문서 + 보조 문서 (몇 개)
```

### 단계 2: 첫 산출물 작성 전 5분 검색 (10분)

```
- 핵심 가정 2~3개 식별
- 각 가정을 흔들 수 있는 데이터 1건씩 검색
- 검색 결과를 첫 산출물의 가이드로 활용
```

이 단계가 v1 → v2 비효율 방지의 핵심.

### 단계 3: 첫 산출물 작성 — 신뢰도 표기 적용 (2~4시간)

```
- 모든 데이터·가정에 🟢/🟡/🔴 표기
- 회사 실재성 vs 행동 검증 분리 (페르소나의 경우)
- 보수적 점수 (낮은 신뢰도부터 시작)
- 본문 + 향후 검증 과제 명시
```

### 단계 4: 셀프리뷰 1차 (30~60분)

```
- 비판자 시각으로 산출물 점검
- 약점 7~11개 도출
- 즉시 해결 가능 vs 인터뷰 영역 분리
- 즉시 해결만 작업 (옵션 B)
```

### 단계 5: 즉시 해결 작업 (1~2시간)

```
- 셀프리뷰의 즉시 해결 가능 약점 처리
- 새 검증 자료 1~2건 추가
- 점수 정직 정정 (인플레이션 해소)
- v2 산출물 완성
```

### 단계 6: 셀프리뷰 2차 (30분)

```
- v2 점검
- 새 약점이 "본질 vs 형식" 비교
- 본질이면 한 번 더 사이클, 형식이면 종료
```

### 단계 7: 마무리 — 종료 선언 + 인터뷰 액션 플랜 (1시간)

```
- 미해결 항목 = PoC 단계 검증 과제로 분리
- 인터뷰 액션 플랜 작성 (얻을 것/보완할 것/결과)
- README + 1pager 작성
- 활용 가이드 (어디서 어떻게 쓸지)
```

## 7. AI(Claude) 페어 기획의 효율화

본 프로젝트는 Claude를 페어 PM으로 활용. AI 페어 기획의 효율화 방법.

### 7.1 AI에게 위임 가능한 것 (효율적)

- 신뢰도 표기 시스템 적용
- 셀프리뷰 비판 자체 도출
- 검색 + 1차 출처 확인
- 문서 압축 + 구조 재정비
- 페르소나 간 정합성 점검

### 7.2 AI에게 위임 불가한 것 (사용자 필수)

- 프로젝트 방향성 결정 (포트폴리오 vs 실제 사업)
- 셀프리뷰 진행 결정 (언제 더 / 언제 멈출지)
- 옵션 A·B·C 선택 (시간·깊이 트레이드오프)
- *"포트폴리오 vs 실제 사업"* 톤 조정
- 최종 산출물 검수

### 7.3 효과적 프롬프트 패턴

본 프로젝트에서 효과 컸던 사용자 프롬프트:

1. *"셀프리뷰해"* — 자체 비판 트리거
2. *"검증 자료 링크 첨부해"* — 1차 출처 강제
3. *"실제 사업 시작 시 동일 프로세스 재사용"* — 톤 결정
4. *"점수 정직 표기"* — 인플레이션 해소
5. *"인터뷰로 뭘 얻으면 뭘 보완해서 어떤 결과를 얻을 수 있다"* — 액션 플랜 강제

이 5가지 프롬프트 패턴이 본 프로젝트의 핵심 동력.

### 7.4 비효율 프롬프트 패턴

본 프로젝트에서 비효율적이었던 패턴:

1. *"더 자세히"* — 방향 없는 확장 → 1,583줄 같은 통제 불능 산출
2. *"리뷰해줘"* (목적 없이) — 무한 루프 위험
3. *"옵션 A로 가자"* (조건 없이) — 시간·자원 검토 없음

다음 기획 시 *"조건부 명령"* 으로 정밀화: *"30분 안에 / 5%p 차이 이내 / 인터뷰 영역 외에서"* 같은 제약 명시.

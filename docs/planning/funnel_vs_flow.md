# Funnel vs Flow Module — 모듈 상세 비교

> 본 문서는 `product_strategy.md` 섹션 11 제품 모듈 구조의 상세 자료다. TryOps의 두 모듈이 같은 Core Platform 위에서 어떻게 분기되며, 입력·처리·출력·UI·가격·영업 시나리오가 어떻게 다른지 정밀 정의.

---

## 0. 두 모듈의 본질적 차이

| 항목 | **Funnel Module** | **Flow Module** |
|---|---|---|
| **목적** | RFID 데이터 + CSI = "왜 안 팔렸나" 진단 | CSI 단독 = "공간 흐름" 분석 |
| **대상 매장** | RFID 도입 완료 매장 | RFID 미도입 매장 (또는 RFID 통합 안 한 매장) |
| **본사 결재 의사결정자** | 본사 데이터팀장 + MD | 본사 운영팀 + 매장 매니저 |
| **수익 모델** | 메인 수익원 | SAM 확보 + 진입 도구 |
| **가격대 (매장당 월)** | 9만~15만원 (Pro), ARR 패키지 | 5만원 |
| **마진** | 30~50% (Pro 기준) | 손실 (한계비용 7.5만원 < 가격 5만원) |
| **영업 사이클** | 8~13개월 (대기업) / 6~9개월 (K-D2C) | 6~9개월 |
| **첫 본사 영업 대상** | 에프앤에프·삼성물산·코오롱FnC 등 RFID 5곳 | 안다르·젝시믹스·마뗑킴 등 K-D2C |

**핵심 명제 (시장 분석 검증 결과 — v3.4)**:
> Flow Module은 단순 *"확장 모듈"* 이 아니라 **K-D2C 본사 영업의 메인 진입 도구**. 한국 패션 RFID 도입 기업이 약 5~10개뿐인 시장 현실에서, Funnel만으로는 SAM이 너무 좁다 (50~100매장). Flow를 통해 K-D2C 본사 진입 → 본사가 RFID 도입 → Funnel 업그레이드의 GTM 사이클이 사업 핵심.

## 1. 입력 데이터 차이

### 1.1 Funnel Module 입력

```
[RFID Adapter]
  └ SKU 식별: 어떤 상품이 피팅룸에 들어가고 나왔는지
  └ 사이즈·색상·재질 메타데이터
  └ 입출고 타임스탬프
  └ 매장 위치 (피팅룸 #1·#2·#3)

[POS Adapter]
  └ 결제 이벤트: 어떤 상품이 결제됐는지
  └ 결제 타임스탬프, 결제 수단, 객단가
  └ 익명 회원 식별 (선택, 가명처리)

[CSI Sensor]
  └ WiFi 신호 변화 데이터 (3~5GHz)
  └ 활동 강도 점수 (raw)
  └ 체류 시간, 움직임 패턴
```

세 데이터 소스의 **시간축 결합**이 Funnel Module의 본질. Try-on session 단위로 통합.

### 1.2 Flow Module 입력

```
[CSI Sensor]
  └ WiFi 신호 변화 데이터
  └ 활동 강도 점수
  └ 체류 시간, 움직임 패턴

[POS Adapter] (선택)
  └ 매장 시간대별 매출 합계 (집계만)
  └ SKU 단위 데이터 없음
```

RFID 없음. **CSI만으로 매장 공간 흐름 분석**.

### 1.3 데이터 처리 깊이 차이

| 처리 단계 | Funnel | Flow |
|---|---|---|
| 매장 게이트웨이 raw 수집 | ✅ | ✅ |
| Session matching (RFID-CSI 시간 결합) | ✅ | ❌ (RFID 없음) |
| SKU-session 단위 hesitation 계산 | ✅ | ❌ |
| 매장 단위 공간 흐름 분석 | ✅ (보조) | ✅ (메인) |
| POS와의 funnel 연결 | ✅ (메인) | ❌ |

## 2. 출력 분석 차이

### 2.1 Funnel Module — Joint Signal 5종

상세는 `product_strategy.md` 섹션 10 참조. 핵심 요약:

| Joint Signal | 정의 | 본사 활용 |
|---|---|---|
| **Hesitation Score** | 같은 SKU/사이즈 반복 입출 + 긴 체류 | MD 의사결정 (가격 인하 vs 사이즈 가이드 개선) |
| **Assistance Need Signal** | 입실 후 일정 시간 + 추가 요청 없음 | 매장 직원 응대 타이밍 알람 |
| **Companion Effect** | 동행자 존재 가능성 신호 | 동행 매장 운영 분석 |
| **Fitting Friction Score** | RFID 회전율 + CSI 대기 강도 | 피크 시간대 피팅룸 운영 개선 |
| **Phantom Try-on Detection** | CSI 입실 + RFID 이벤트 없음 | RFID 데이터 품질 검증 |

### 2.2 Flow Module — 공간 흐름 5종

| Flow Signal | 정의 | 본사 활용 |
|---|---|---|
| **Occupancy Pattern** | 매장 시간대별 점유율 | 피크 시간·요일 분석 |
| **Wait Duration** | 피팅룸 대기 시간 추정 | 대기 병목 진단 |
| **Dwell Time** | 매장 평균 체류 시간 | 매장 매력도 평가 |
| **Rotation Rate** | 피팅룸 회전율 | 매장 운영 효율 |
| **Anomaly Detection** | 비정상 매장 상태 (방치·과밀) | 운영 위험 알람 |

Joint Signal의 *"SKU·session 정밀도"* 가 없고, 매장 단위 일반 흐름만 측정.

## 3. UI / 대시보드 차이

### 3.1 Funnel Module UI

**본사 데이터팀장 (김지현·박정우) 대시보드**:
- High try-on / low conversion SKU 리스트
- 매장 간 conversion gap 시각화
- 사이즈별 Hesitation 분포
- 시즌별 SKU 실패 원인 후보 분류

**본사 머천다이저 (박서연) 대시보드**:
- 본인 담당 SKU 80~120개 일별 funnel
- 매장 매니저 정성 코멘트 vs 정량 데이터 매칭
- A/B 테스트 결과 (사이즈 가이드 수정 전후)

**매장 매니저 (정민호) 대시보드**:
- 매장 단위 실시간 funnel
- Assistance Need 알람 (피팅룸 #3, 7분 경과)
- Phantom 후보 알림 (RFID 누락 의심)

### 3.2 Flow Module UI

**본사 운영팀 대시보드**:
- 매장별 시간대별 점유율 히트맵
- 매장 간 dwell time 비교
- 이상 상태 알람 (과밀·방치)

**매장 매니저 대시보드**:
- 매장 실시간 흐름 모니터
- 피크 시간 예측 알람
- 매장 운영 KPI 일별 트렌드

### 3.3 UI 공유 vs 분기 구조

```
Core Platform UI
├─ 공유 (70%): 매장 관리·권한·설정·알람·계정
├─ Funnel 전용 (20%): Joint Signal 5종, SKU-session 마트
└─ Flow 전용 (10%): 공간 흐름 5종, 매장 흐름 마트
```

UI 코드 공유율 70%로 개발·유지보수 효율 확보.

## 4. 가격 정책 차이

### 4.1 가격 구조

| 플랜 | 모듈 | 대상 | 가격 (매장당 월) |
|---|---|---|---|
| **Flow Standalone** | Flow | RFID 미도입 매장 | 5만원 |
| **Funnel Starter** | Funnel | 20매장 이하 본사 | 9만원 |
| **Funnel Pro** | Funnel | 21~100매장 본사 | 15만원 |
| **Funnel Enterprise ARR** | Funnel | 100매장 이상 대기업 | 매장당 8~12만원 환산 (본사 ARR 1.5억~3억원/년) |

**일회성 도입비**: 매장당 약 30만원 (노드 4~6개 + 게이트웨이 + 설치)

### 4.2 Unit Economics

매장당 한계 비용 (cloud + 운영): **약 7.5만원/매장/월**
- Kinesis on-demand: 1만원
- S3 + Athena: 2만원
- 매장 게이트웨이 (감가 1/60): 0.5만원
- 노드 6개 (감가 1/60): 1만원
- 영업/CS 인건비 분산 (100매장 운영 시): 3만원

| 플랜 | 가격 | 마진 | 마진율 | 의미 |
|---|---|---|---|---|
| Flow Standalone | 5만원 | -2.5만원 | **손실** | SAM 확보 전략 |
| Funnel Starter | 9만원 | 1.5만원 | 16.7% | 신규 본사 락인 (미끼 가격) |
| Funnel Pro | 15만원 | 7.5만원 | 50% | **메인 수익원** |
| Funnel Enterprise ARR | 8~12만원 | 0.5~4.5만원 | 6~38% | 본사 ARR 일괄 계약 |

### 4.3 본사당 ARPU 가정

| 본사 규모 | 매장 수 | 모듈 조합 | 본사당 ARR |
|---|---|---|---|
| 소형 K-D2C | 5~10매장 | Flow 50% + Funnel Starter 50% | 약 800만~1,500만원 |
| 중형 K-D2C | 20~30매장 | Funnel Pro 100% | 약 3,600만~5,400만원 |
| 대기업 | 100매장+ | Funnel Enterprise ARR | 약 1.5억~3억원 |

본사당 평균 ARPU **매장당 12만원 이상** 유지가 사업성 핵심.

🔴 ARR 패키지 결론은 박정우 페르소나 행동 검증 30% 수준 기반. PoC 단계 대기업 영업 시 실측 필요.

## 5. 영업 시나리오 차이

상세는 `personas_extended.md` v4 섹션 8 영업 사이클 시간 분해 참조.

### 5.1 시나리오 A — Funnel Module (대기업)

**1차 대상**: 에프앤에프 (디스커버리·MLB), 삼성물산 패션 (에잇세컨즈), 코오롱FnC, 크리스에프앤씨 (골프웨어), 한세엠케이 (NBA·TBJ)

**영업 사이클**: 8~13개월 (`personas_extended.md` 시간 분해)

**핵심 영업 메시지**:
> *"이미 도입하신 RFID에 행동 신호를 더해 RFID ROI를 강화합니다. SAP S/4 HANA와 webhook 통합되고, 매장 CCTV가 못하는 본사 통합 분석이 가능합니다. MLB 미국 매장과 디스커버리 한국 매장의 행동 데이터를 같은 대시보드에서 보십시오. 첫 도입 6개월, ROI 검증 12개월의 단계별 PoC 패키지를 제안드립니다."*

**거부형 페르소나**:
- 김상철 (본부장 거부형, `personas_extended.md` 섹션 2)
- 강희정 (매장 매니저 거부형, 섹션 3)

**거부 우회 전략**:
- 박정우 (데이터팀장) 챔피언화 → 본부장 우회
- 동료 본부장 사례·6개월 ROI 시뮬레이션
- 매장 매니저 본인 대시보드 (방패 데이터)

### 5.2 시나리오 B — Flow Module (K-D2C)

**1차 대상**: 안다르, 젝시믹스, 마뗑킴, 마르디 메크르디, 비비안

**영업 사이클**: 6~9개월

**핵심 영업 메시지**:
> *"글로벌 매장 확장 시점에 매장 데이터 표준화의 도구입니다. 피팅룸 안의 행동 신호를 영상 없이 잡아드립니다. RFID 미도입 매장도 즉시 적용 가능합니다. 향후 RFID 도입 시 Funnel Module로 업그레이드 가능합니다."*

**거부형 페르소나**:
- 이수정 (K-D2C 마케팅 본부장 거부형, 섹션 4)

**거부 우회 전략**:
- TryOps는 *"마케팅 영역 무관"* 차별화 (이미 컨설팅 쓰고 있는 본부장 안심)
- 데이터팀 예산으로 추진 (마케팅 예산과 분리)
- 김지현 (데이터팀장) 단독 결재 가능성 활용

### 5.3 GTM 사이클 — Flow → Funnel 업그레이드

```
Year 1: K-D2C 본사 영업 (Flow Module)
  └ 안다르·젝시믹스·마뗑킴 등 3~5곳 도입
  └ 매장당 월 5만원 (손실 사업, 진입 도구)
  └ 시장 점유 확보 (SAM 활용)

Year 2: 본사가 RFID 도입 결정
  └ TryOps의 권유 또는 자체 결정
  └ Flow Module 매장 → RFID 추가 설치
  └ Funnel Module 자동 업그레이드 가능

Year 3: Funnel Module 메인 수익화
  └ 본사당 ARPU 5만 → 15만원 (3배 증가)
  └ 매장당 마진 -2.5만 → 7.5만원 (손실 → 50% 마진)
  └ 사업성 본격 확보
```

이게 *"Flow는 손실 사업"* 의 정당성. 1~2년 손실 감수 + 3년차 수익 회복의 GTM.

### 5.4 영업 메시지 확장 — 거부 대응 스크립트

본사 영업 시 자주 받는 7가지 거부 질문과 대응 스크립트.

**Q1: *"RFID 데이터만으로도 분석 가능한데 굳이 CSI?"***
> *"RFID는 '무엇이 들어갔다'까지 봅니다. CSI는 그 안에서 망설였는지, 반복 비교했는지, 대기·마찰이 있었는지를 봅니다. 두 데이터의 결합이 'high try-on / low conversion 상품의 원인 후보 발견'이라는 차별점을 만듭니다. PoC 4주에 RFID-only로는 안 보이고 TryOps로는 보이는 케이스를 5개 이상 보여드립니다."*

**Q2: *"이미 매장 CCTV가 있는데 동선 분석은 충분하지 않나?"***
> *"매장 CCTV는 피팅룸 내부를 못 봅니다 — 개인정보보호법 제25조에 의해 탈의실 영상 촬영 금지입니다. 피팅룸은 손님 구매 결정의 핵심 구간이지만 영상 분석으로는 진입 불가능한 영역입니다. TryOps는 비영상 신호로 이 영역을 합법적으로 분석할 수 있는 유일한 솔루션입니다."*

**Q3: *"개인정보 문제 안 생기나?"***
> *"본 시스템은 카메라·음성·얼굴·신체 이미지·MAC 주소를 일절 수집하지 않습니다. WiFi 신호의 공간 변화만 측정하며, 개인 식별 안 됩니다. 개인정보보호법 영상정보처리기기 규제에서 면제됩니다. 한국 법무법인 정식 의견서 제공 가능합니다."*

**Q4: *"한국 스타트업 제품의 안정성 우려"***
> *"PoC 4주는 무료로 진행해드립니다. 그 단계에서 안정성·정확도 확인 후 확장 도입 결정하시면 됩니다. PoC 비용은 본사 부담 0원입니다."*

**Q5: *"매장 운영 부담은?"***
> *"매장 노드 설치는 1시간, 매장 직원 추가 작업은 0입니다. 매장 매니저 본인 대시보드도 제공해드려서 본사 압박에서 본인을 방어할 수 있는 데이터로 활용 가능합니다."*

**Q6 (이수정 결): *"이미 마케팅 컨설팅 받고 있는데..."***
> *"TryOps는 매장 운영 데이터입니다. 마케팅 컨설팅의 영역과 겹치지 않습니다. 데이터팀 예산으로 도입 가능하며, 기존 마케팅 분석과 별개의 새로운 인사이트를 제공합니다."*

**Q7 (김상철 결): *"RFID도 활용 못 하는데 또 다른 도구?"***
> *"바로 그 이유로 TryOps가 필요합니다. RFID의 ROI를 살리는 도구입니다. 이미 도입하신 RFID 데이터에 행동 신호를 더해서 'RFID 도입 ROI'를 임원 보고용 데이터로 만들어드립니다. PoC 4주에 RFID 데이터 활용도가 어떻게 변하는지 보여드립니다."*

이 7가지 스크립트가 영업 사이클 8~13개월(시나리오 A) / 6~9개월(시나리오 B) 중 가장 빈번하게 사용되는 거부 대응 자료.

## 6. 기술 스택 공유 vs 분기

### 6.1 공유 컴포넌트 (코드 공유율 70%)

```
[Core Platform]
├ ESP32-S3 펌웨어 (CSI 수집)
├ 매장 게이트웨이 (raw 데이터 수집·송신)
├ Kinesis Streams (실시간 적재)
├ S3 + Athena (가공 데이터 마트)
├ 매장 관리·권한·설정 UI
├ 알람 시스템
├ 멀티테넌트 IAM
└ 본사·매장 대시보드 공유 컴포넌트
```

### 6.2 Funnel 전용 모듈

```
[Funnel-only]
├ RFID Adapter (vendor별 — 셀메이트·idro·하이태그랩)
├ POS Adapter (vendor별 — 토스플레이스·NHN KCP·NICE 등)
├ Session Matching Engine (RFID-CSI 시간 결합)
├ Joint Signal 계산 엔진 (5종)
├ SKU-session 마트
├ High try-on / low conversion 분석
└ Funnel 전용 대시보드
```

### 6.3 Flow 전용 모듈

```
[Flow-only]
├ Spatial Flow Analytics Engine
├ Anomaly Detection
├ 매장 단위 흐름 마트
└ Flow 전용 대시보드
```

### 6.4 기술 스택 진화 (Phase별)

| Phase | 데이터 처리 스택 | 모듈 |
|---|---|---|
| Phase 1·2 | Python + Polars + DuckDB + Airflow + S3 Parquet + PostgreSQL | Funnel + Flow 통합 |
| Phase 3 | PySpark on EMR Serverless (매장 200개 임계점) | Funnel + Flow 분기 |
| Phase 4 | Flink streaming (실시간 알람 본격화) | Funnel 실시간 |

Polars API 패턴이 Spark와 호환되도록 처음부터 설계. 매장 200개 도달 시 자연스러운 마이그레이션.

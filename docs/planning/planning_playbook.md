# Planning Playbook — 기획 시작 시 펼쳐서 쓰는 실행 가이드

> 본 문서는 *"다음에 기획할 때 더 적은 노력으로 더 완성된 값을 낼 수 있게"* 의 실용 매뉴얼이다.
>
> 본 TryOps 프로젝트는 약 30~40시간 소요됐고, 시행착오 끝에 검증 시스템·셀프리뷰·인터뷰 액션 플랜의 골격을 확보했다. 다음 기획은 본 가이드로 **약 8~12시간 (30% 시간)** 에 동일 깊이 산출 가능.
>
> 본 문서는 *"읽는 문서"* 가 아니라 *"기획 중 옆에 펼쳐두고 체크하는 문서"* 다. 회고 분석은 `lessons_learned.md` 참조.

---

## 0. 시작 전 30초 — 본 가이드의 사용법

1. 새 기획 프로젝트 시작 직전 본 문서를 펼친다
2. **섹션 1 (시작 선언 템플릿)** 을 채우고 시작
3. 작업 중 **섹션 2 (7단계 워크플로우)** 를 따라간다
4. 단계마다 **섹션 3 (체크리스트)** 로 점검
5. 막힐 때 **섹션 4 (의사결정 가이드)** 참조
6. 종료 시 **섹션 5 (마무리 템플릿)** 작성

---

## 1. 시작 선언 템플릿 (작업 시작 전 30분 필수)

새 프로젝트 시작 시 다음 5가지를 명시적으로 결정한다. *"하면서 결정"* 은 비효율을 만든다.

```yaml
프로젝트명: 
한 줄 정의: 

목표:
  □ 포트폴리오 (면접·취업용)
  □ 실제 사업 시작 (영업·법무·자금 자료)
  □ 학습용 (PM 사고 훈련)
  
검증 깊이:
  □ 공개 자료만 (1차 출처 30~50건 목표)
  □ 인터뷰 포함 (PoC 단계 진입, 인터뷰 50건+)
  □ 실측 포함 (실제 환경 데이터 수집)

종료 조건 (셀프리뷰 무한 루프 방지):
  □ 3사이클 후 종료
  □ 셀프리뷰 점수 차이 5%p 이내 시 종료
  □ 인터뷰·실측 영역 진입 시 종료 (공개 자료 한계점)
  □ 새 약점이 "본질 → 형식"으로 이동 시 종료

산출물 구조 (사전 정의):
  메인 문서:
  보조 문서: 
    - 
    - 
    - 

핵심 가정 2~3개 (시작 전 5분 검색 대상):
  1. 
  2. 
  3. 
```

이 템플릿을 **종이에 적든 별도 메모에 적든** 의식적으로 채운다. 모호하게 시작하면 1,583줄 같은 통제 불능 산출물이 나온다.

## 2. 7단계 워크플로우 (총 8~12시간)

### 단계 1: 프로젝트 시작 선언 (30분)

- 섹션 1 템플릿 작성
- 산출물 폴더 생성
- **첫 메인 문서의 *"섹션 구조"* 사전 정의** (10개 섹션 정도)

**중요**: 이 단계를 스킵하면 5~10시간 비효율 발생. 30분 투자가 본 워크플로우의 최대 ROI.

### 단계 2: 5분 검색 (10분)

핵심 가정 2~3개를 흔들 수 있는 데이터를 검색.

**검색 패턴**:
```
1. "[핵심 가정] + 한국 + [최신 연도]" 키워드 검색
2. 1차 출처 (통계청·KPMG·언론사·학술지) 우선 선택
3. 가정이 흔들리는 데이터가 있으면 메모
```

**예시**:
- 가정: *"K-D2C 본사가 데이터 도구 도입에 적극적"*
- 검색: *"K-D2C 본사 디지털 전환 한국 2025"*
- 발견: HBR *"변화 4분의 3 실패"* → 가정 즉시 보수화 필요

이 5분이 v1 → v2 비효율을 막는 핵심.

### 단계 3: 첫 산출물 작성 (2~4시간)

**작성 원칙 4가지**:

```
1. 신뢰도 표기 (🟢/🟡/🔴) 모든 데이터·가정에 적용
2. 회사/조직 실재성 vs 행동 검증 분리 표기 (페르소나의 경우)
3. 보수적 점수 (낮은 신뢰도부터)
4. 검증 자료 인용 시 "맥락" 명시 (어떤 조건의 데이터)
```

**섹션 구조 가이드 (메인 문서)**:
- 섹션 0: Executive Summary (30초 자료)
- 섹션 1~3: 문제 정의·Why Now·ICP
- 섹션 4~5: 페르소나·의사결정 회의
- 섹션 6~7: 신뢰 설계·시장 분석
- 섹션 8~9: 법적 검토·제품 모듈
- 섹션 10~11: 가격 정책·ROI
- 섹션 12: MVP·PoC
- 섹션 13: 실패 시나리오
- 섹션 14~15: 학습 로드맵·최종 포지셔닝

각 섹션 100~150줄 목표 → 메인 문서 약 1,000줄. **이걸 넘으면 보조 문서 분리**.

### 단계 4: 셀프리뷰 1차 (30~60분)

**셀프리뷰 진행 방법**:

```
1. 본인 산출물을 "면접관 5분 시각" + "동료 1시간 시각" 으로 점검
2. 약점 7~11개 도출 (강점은 따로 정리, 약점 집중)
3. 약점 분류:
   - 즉시 해결 가능: 다음 단계 작업 대상
   - 인터뷰·실측 영역: PoC 단계 분리
   - 형식적 비판: 무시
```

**셀프리뷰 트리거 질문 (Claude 또는 본인 비판자 시각)**:

- *"이 데이터가 진짜로 가정을 입증하는가?"*
- *"점수가 인플레이션 됐을 가능성?"*
- *"검증 자료의 맥락이 정확한가?"*
- *"가설이 흔들리는 데이터를 무시했나?"*
- *"면접관이 의심할 부분은 어디?"*
- *"이 산출물이 실제 사업에 어떻게 쓰이는가?"*

### 단계 5: 즉시 해결 작업 (1~2시간)

셀프리뷰의 *"즉시 해결 가능"* 약점만 처리. **인터뷰 영역은 분리하고 작업 안 함**.

**작업 패턴**:
- 새 검증 자료 1~2건 추가 (반증 데이터 우선)
- 점수 정직 정정 (인플레이션 해소)
- 톤 조정 (메타 평가 자기 만족 제거)
- 가설 흔들림이 발견되면 정면 보고

**점수가 떨어지는 게 정직성 향상** — 이 원칙을 잊지 말 것.

### 단계 6: 셀프리뷰 2차 (30분)

**판단 기준**:

```
새 약점의 종류 점검:
- 본질적 (검증·인과·가설) → 한 번 더 사이클 (단계 5 반복)
- 형식적 (표기·구조·정합성) → 종료
- 인터뷰·실측 영역 → PoC 단계로 분리, 종료
```

이 판단이 **셀프리뷰 무한 루프 방지의 핵심**. *"새 약점이 본질에서 형식으로 이동"* 하는 순간 종료.

### 단계 7: 마무리 (1시간)

마무리 산출물 3종:

**산출물 1: README** — 프로젝트 인덱스
- 5분 / 15분 / 1시간 읽기 가이드
- 문서 간 연결 도식
- 면접 Q&A 5~7개
- 검증 시스템 소개

**산출물 2: 1pager** — 30초 자료
- 한 줄 정의 + 핵심 명제
- 핵심 수치 표
- 결정적 차별점
- 가장 큰 리스크 3가지
- 한 줄 영업 메시지

**산출물 3: 인터뷰 액션 플랜** — 미해결 항목의 PoC 분리
- 인터뷰 대상 우선순위 (P0~P3)
- 인터뷰당: 얻을 것 → 보완할 것 → 결과 → 비용 → 시점
- 인터뷰 후 산출물 (어떤 신규 문서 작성될지)

## 4. 의사결정 가이드 — 막힐 때 참조

### 4.1 *"문서 너무 길어지는데 분리해야 하나?"*

```
판단 기준:
- 메인 문서 1,000줄 초과 → 보조 문서 분리
- 한 섹션이 200줄 초과 → 보조 문서 분리
- 보조 문서 후보가 5개 이상 → 우선순위 3개만 작성 (Phase 1)
- 나머지 → "Phase 2" 또는 "향후 작성" 으로 명시
```

### 4.2 *"검증 자료 더 찾아야 하나, 멈춰야 하나?"*

```
판단 기준:
- 핵심 가정 2~3개에 1차 출처 1~2건씩 있으면 충분
- 추가 검색이 새 인사이트 안 줌 → 멈추기
- 가정이 흔들리는 데이터 발견 → 가정 정정 후 멈추기
- 검색 5분 넘어도 데이터 없음 → "PoC 단계 검증 과제"로 분리
```

### 4.3 *"셀프리뷰 더 해야 하나, 멈춰야 하나?"*

```
판단 기준:
- 새 약점이 본질적 (검증·인과·가설) → 한 번 더
- 새 약점이 형식적 (표기·구조·정합성) → 멈추기
- 새 약점의 해소 방법이 인터뷰뿐 → 멈추기 (PoC 분리)
- 점수 차이가 5%p 이내 → 멈추기
- 3사이클 도달 → 멈추기
```

### 4.4 *"점수를 몇 점으로 매겨야 하나?"*

```
신뢰도 점수 가이드:
🟢 검증됨 (1차 출처 직접 인용): 단일 점수 부여 가능
🟡 추정 (검증 데이터 기반 계산): 보수적 점수
🔴 가설 (검증 안 됨): 명시적으로 "가설" 표기

페르소나 점수 분리:
- 회사/조직 실재성: IR·채용 공고·언론 검증 가능
- 페르소나 행동 검증: 인터뷰 없이는 낮을 수밖에 없음
- 단일 점수 매기지 말 것

원칙: 점수가 낮아질수록 정직성 향상
- 첫 표기 보수적으로 시작
- 인플레이션 발견 시 즉시 정정
- "검증 못 했다"고 명시하는 게 진짜 검증
```

### 4.5 *"가설이 검증으로 흔들렸는데 어떻게 해야 하나?"*

```
가설 흔들림 발견 시:
1. 절대 숨기지 말 것 (가장 큰 PM 사고 실수)
2. "수정 사항"으로 명시
3. 흔들린 가설 vs 새 가설을 표로 비교
4. 새 가설도 가설임을 명시 (인터뷰 검증 필요)
5. 새 가설이 비즈니스 전략에 미치는 영향 정직 보고

예시: TryOps의 95:5 vs 76.2% 발견
- v3에서 정정 명시
- 신뢰 설계 우선순위 재설계
- 본 발견 자체를 강점으로 제시 (PM 사고의 정수)
```

### 4.6 *"인터뷰 액션 플랜은 어떻게 작성하나?"*

```
인터뷰별 5요소 매핑:
1. 무엇을 얻나 (정량/정성 인사이트)
2. 무엇을 보완하나 (어떤 문서·결정 업데이트)
3. 얻을 결과 (페르소나 검증율 변화 + 사업 의사결정)
4. 인터뷰 방법 (콘택트 경로 + 시간 + 비용)
5. 우려점 (인터뷰의 한계)

우선순위 매트릭스:
- P0 (긴급): 가설 흔들 가능성 큰 인터뷰
- P1 (핵심): 영업 사이클 시작 위한 인터뷰
- P2 (중요): 사용자 페르소나 정밀화
- P3 (보강): 거부형 페르소나·경쟁사 검증

비용 추정:
- 손님 인터뷰: 5,000원/건 인센티브
- 본사 인터뷰: 비공식 식사 30~50만원
- 임원 인터뷰: 30~50만원
- 솔루션사 인터뷰: 20~30만원
```

### 4.7 *"AI(Claude)에게 어떻게 시켜야 효율적인가?"*

```
AI에게 위임 가능 (효율적):
- 신뢰도 표기 시스템 적용
- 셀프리뷰 비판 자체 도출
- 검색 + 1차 출처 확인
- 문서 압축 + 구조 재정비
- 페르소나 간 정합성 점검

AI에게 위임 불가 (본인 필수):
- 프로젝트 방향성 결정 (포트폴리오 vs 실제 사업)
- 셀프리뷰 진행 결정 (언제 더 / 언제 멈출지)
- 옵션 A·B·C 선택
- 톤 조정
- 최종 산출물 검수

효과적 프롬프트 패턴:
1. "셀프리뷰해" → 자체 비판 트리거
2. "검증 자료 링크 첨부해" → 1차 출처 강제
3. "[목표] 위해 [톤] 으로" → 방향 명시
4. "점수 정직 표기" → 인플레이션 해소
5. "인터뷰로 뭘 얻으면 뭘 보완해서 어떤 결과를 얻을 수 있다" → 액션 플랜 강제

비효율 프롬프트 회피:
- "더 자세히" (방향 없는 확장)
- "리뷰해줘" (목적 없는 무한 루프)
- "옵션 A로" (조건 없는 선택)

조건부 명령으로 정밀화:
- "30분 안에 / 5%p 차이 이내 / 인터뷰 영역 외에서" 같은 제약 명시
```

## 5. 마무리 템플릿

프로젝트 종료 시 다음 3가지를 작성하고 종료 선언.

### 5.1 README 핵심 구조

```markdown
# 프로젝트명 — 한 줄 정의

## 0. 본 프로젝트가 무엇인가
- 목표 + 톤 (포트폴리오/실제 사업)
- 본 프로젝트가 한 것 vs 안 한 것

## 1. 5분/15분/1시간 읽기 가이드

## 2. 문서 N개의 역할과 연결 (도식)

## 3. 핵심 메시지 — 30초 버전

## 4. 검증 시스템 (🟢/🟡/🔴 + 분리 + 셀프리뷰)

## 5. 포트폴리오 vs 실제 사업 톤 명시

## 6. 면접 Q&A 5~7개

## 7. 미해결 한계 (정직 명시)

## 8. 본 프로젝트로 얻은 메타 학습

## 9. 활용 가이드 (어디서 어떻게 쓸지)
```

### 5.2 1pager 핵심 구조

```markdown
# 프로젝트명 — 1pager

## 한 줄 정의

## Beach-head 영업 대상 N곳

## 가장 큰 리스크 3가지 + 대응

## 한 줄 영업 메시지

## 다음 단계 — PoC 1년차 (예산 + 일정)
```

### 5.3 인터뷰 액션 플랜 핵심 구조

- 🔴 [항목 1]
- 🔴 [항목 2]
- ...

본 프로젝트 최종 산출물:
- 문서 N개, 총 N줄, 1차 출처 N건
- 평균 검증율 N% (보수적 표기)

## 6. 시간 추정 가이드

본 워크플로우의 단계별 시간 추정:

| 단계 | 최소 시간 | 권장 시간 | 비고 |
|---|---|---|---|
| 1. 시작 선언 | 20분 | 30분 | 스킵하면 5~10시간 비효율 발생 |
| 2. 5분 검색 | 5분 | 10분 | 핵심 가정 2~3개에 집중 |
| 3. 첫 산출물 | 2시간 | 4시간 | 메인 문서 1,000줄 목표 |
| 4. 셀프리뷰 1차 | 30분 | 60분 | 약점 7~11개 도출 |
| 5. 즉시 해결 | 1시간 | 2시간 | 새 자료 1~2건 추가 |
| 6. 셀프리뷰 2차 | 20분 | 30분 | 종료 판단 |
| 7. 마무리 | 30분 | 1시간 | README + 1pager + 인터뷰 플랜 |
| **합계** | **4.5시간** | **8.7시간** | TryOps 30~40시간의 약 25% |

**+α 옵션** (필요 시):
- 보조 문서 분리 (시간당 1개): 우선순위 3개부터
- 추가 셀프리뷰 사이클 (사이클당 2~3시간)
- 회고 작성 (1시간): 다음 프로젝트 학습 누적

총 12~15시간으로 본 TryOps의 30~40시간 깊이 달성 가능.

## 7. 자주 하는 실수와 회피법

### 실수 1: 시작 선언 스킵

**증상**: *"일단 작성하면서 결정하자"*  
**결과**: v2 1,583줄 같은 통제 불능 산출물  
**회피**: 섹션 1 템플릿 30분 무조건 작성

### 실수 2: 검증 자료 없이 작성

**증상**: *"일단 가설로 쓰고 나중에 검증"*  
**결과**: v2 → v3 재작성 필요, 시간 2배  
**회피**: 단계 2의 5분 검색 무조건 수행

### 실수 3: 점수 인플레이션

**증상**: *"우리 페르소나는 70% 검증됐다"*  
**결과**: 셀프리뷰에서 정정 작업 반복  
**회피**: 첫 표기를 보수적으로. *"30% 검증, 70% PoC 필요"* 가 정직

### 실수 4: 셀프리뷰 무한 루프

**증상**: 매 사이클 새 약점 발견, 끝없이 진행  
**결과**: 시간 소모, 본질에서 멀어지는 비판 누적  
**회피**: 단계 6의 종료 판단 명확화. 새 약점이 형식적이면 종료

### 실수 5: 가설 흔들림 숨기기

**증상**: *"95:5 가정이 76.2%로 흔들렸지만 그냥 두자"*  
**결과**: 면접관·동료가 발견 시 신뢰도 추락  
**회피**: 가설 흔들림 발견 즉시 정면 보고. 본 발견 자체를 강점으로 제시

### 실수 6: 메타 평가 톤

**증상**: 문서 마지막에 *"본 문서가 가진 진짜 가치는..."*  
**결과**: 자기 만족 톤이 신뢰도 떨어뜨림  
**회피**: *"본 문서 활용 가이드: 이걸 어떻게 쓰는가"* 의 사용자 안내 톤

### 실수 7: 보조 문서 너무 많이 만들기

**증상**: 보조 문서 10개+ 작성 계획  
**결과**: 시간 소모, 산출물 분량 인플레이션  
**회피**: 우선순위 3개부터. 나머지는 *"Phase 2 / 향후 작성"* 으로 명시

## 8. 도메인별 특화 팁

### 8.1 B2B SaaS 기획

- 페르소나는 *"메인 3인 + 확장 6인"* 구조 권장
- 거부형 페르소나 필수 (의사결정 차단 가능 인물)
- 영업 사이클 페르소나별 시간 분해
- 가격 협상 매뉴얼 (시나리오별)

### 8.2 B2C 제품 기획

- 손님 페르소나 더 정밀 (3~5인)
- 75:5 같은 단순 비율 가정 회피 (검증 어려움)
- SNS 위기 대응 매뉴얼 필수
- 사용자 경험 (UX) 시나리오 추가

### 8.3 한국 시장 진입 기획

- 한국 법규 검토 필수 (개인정보보호법·의료기기법 등)
- 한국 RFID·결제·매장 인프라 실재 검증
- KOSIS 통계 한계 (검색만으로 정확 수치 추출 어려움)
- 1차 출처: 통계청·KPMG·한국경제·패션비즈·bizwatch 등

### 8.4 글로벌 SaaS 기획

- 글로벌 비교 SaaS 1~2건 필수 (RetailNext·Nexite 등)
- 가격 정책 글로벌 환산 (USD)
- 법규 국가별 차이 (개인정보·근로자·의료)
- 영업 사이클 국가별 차이
